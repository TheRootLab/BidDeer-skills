# BidDeer 标书逐项检查 Skill 输入输出协议 v0.1

该文档用于定义 `proposal-point-checker` Skill 的标准输入与标准输出格式。

**核心定位**：人工提供审核点清单，AI 根据审核点逐项检查投标文件，输出证据状态、标书体现、出处、原文上下文和核对提示。

**注意**：该 Skill 不负责自动判断是否废标，不替代人工复核，不输出最终投标结论。

---

输入模板是给人工填写的审核点清单。请保持极简。

|---:|---|---|---|

### 字段解释

| 字段    | 是否必填 | 说明                        |
| ----- | ---- | ------------------------- |
| 序号    | 必填   | 用于对应 AI 输出结果中的要点编号        |
| 审核点名称 | 必填   | 对审核点的简短命名，例如“项目经理同类项目经验”  |
| 审核要求  | 必填   | 招标文件中的硬性要求、量化要求或关键条件      |
| 审核说明  | 必填   | 告诉 AI 应重点检查什么，以及需要注意的口径差异 |

### 输入示例

| 序号 | 审核点名称 | 审核要求 | 审核说明 |
|---:|---|---|---|
| 1 | 项目经理同类项目经验 | ≥5 年同类项目管理经验 | 检查项目经理是否具备招标文件要求的同类项目管理经验年限，注意区分总工龄、从业年限、项目管理经验和同类项目经验。 |
| 2 | 技术支持响应时间 | 7×24 小时服务，2 小时内响应 | 检查售后服务方案、服务承诺函、运维服务章节中是否体现 7×24 小时支持、响应时间和到场时间。 |
| 3 | 核心软硬件参数响应 | 内存 ≥128GB，存储 ≥4TB | 检查技术标中的核心软硬件配置、性能指标、容量指标是否满足招标要求，并注意与报价清单是否一致。 |
| 4 | 暗标信息检查 | 不得出现投标人名称、Logo、联系人、电话、地址等识别信息 | 如本项目为暗标，检查技术标正文、页眉页脚、图片、截图、附件、文件属性中是否出现可识别投标人的信息。 |

## 2. 证据状态枚举

| 状态 | 含义 | 使用场景 |
|---|---|---|
| ✅ 已定位到明确证据 | 找到的证据与审核要求高度一致，且未发现明显冲突 | 要求为“内存 ≥128GB”，标书体现“内存 256GB” |
| 🟡 已定位到疑似证据 | 找到相关内容，但主体、范围、时间、口径、文件位置等仍存在不确定 | 找到“服务支持”，但未明确是否为本项目服务承诺 |
| 🟠 已定位到冲突证据 | 找到相关内容，但与审核要求存在数字、时间、主体、范围、前后文或跨文件冲突 | 要求“同类经验 ≥5 年”，标书体现“总工龄 8 年 / 同类经验 3 年” |
| 🔴 未定位到证据 | 在已检索范围内未找到能够支撑该审核点的证据 | 未找到响应时间、承诺函、证书或社保证明 |
| ⚪ 证据不足 | 找到部分信息，但不足以完整支撑审核要求 | 找到人员姓名，但缺证书编号、有效期或社保证明 |
| ⚫ 无法判断 | 文件缺失、扫描件不可读、OCR 失败、表格解析失败或审核要求本身不明确 | 附件无法打开、扫描件模糊、表格跨页识别失败 |

#### 总表字段解释

| 字段        | 是否必填 | 说明                               |
| --------- | ---- | -------------------------------- |
| 序号        | 必填   | 对应输入模板中的审核点序号                    |

## 5. 输出约束 (Output Constraints)

1. **结构顺序**：必须先输出检查结果总表，再输出问题项详情。
2. **完整覆盖**：总表必须逐项覆盖所有输入审核点。
3. **强制展开**：所有非“✅ 已定位到明确证据”的项目必须提供详情。
4. **量化对齐**：涉及数字、金额、日期、年限、人数、比例、数量时，必须进行“审核要求 → 标书体现”的对齐。
5. **证据真实性**：不得编造页码、文件名、原文或证据。找到证据时必须输出出处和原文上下文。
6. **检索说明**：未找到证据时，必须说明已检索范围。
7. **结论边界**：发现冲突时，只输出“核对提示”，不得输出最终废标结论。
8. **异常处理**：如文件不可读、缺失、OCR 失败或表格解析失败，应标记为“⚫ 无法判断”。
